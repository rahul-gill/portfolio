Hash tables

Hash tables are data structures for holding key-value pairs of data with O(1)\mathcal{O}(1)O(1) amortized add, delete and find operation. But the order information of keys is lost unlike the balanced tree structures.

When the keys are simple and the space of keys is small, a simple mapping from keys to array indexes is suffice. For example for key-value mapping of student roll numbers and their results. We'll just use roll_no-1 as index for the array in which we store the results. We can access the result of a student in O(1)\mathcal{O}(1)O(1) with accessing arr[roll_no-1]. For deleting, we can put a flag as value for the key, we can put null, -1 or something like that.

When the space of keys is large, for example when keys can have values ranging from 1 to 1e6, we'll need a way to map the keys to array indexes. For that we use hash functions. Although, we can simply do modulo operation by the size of array on the keys to map them to index, but that could result in lots of keys getting the same index, we'll see how hash functions and some techniques help with that.

A hash function returns an interger value, should take just a few cpu cycles for computation and must return the same value for same object. But a hash function can return same value for two object, we call it a collision. Now how do we make use of hash functions to compute indexes to store the object in hash table array? A hash function can also have large values just like original keys, but hash function return values are randomly distributed and are less like to cause collisions. We can't completely elimiate collision so we'll use some techniques to help with that.

A sample hash function for string is given below(java language use this hash function for strings, see here)

int hashCode(string value) {
    int h = 0;
    for(char x: value)
        h = 31 * h + x;
    return h;
}

Here, to account for the position of characters in string, we multiply then by const^char_position the choosen const here is 31. To convert the hash to table index, we use modulo operation(hash % table_size, to optimize this approach we choose only prime numbers sizes of table(approach used by c++,see here) we also make the hash value positive by resetting the sign bit of signed integer. In 2's complement notation, the most signification bit is the sign bit. To reset sign bit we take bitwise and of the number and 0x7F(for 8 bit hash function)

0b0000 0000 -> 0
0b0000 0001 -> 1
0b1000 0000 -> -128
0b1111 1111 -> -1 

0b1000 0000 & 0b0111 1111 = 0b0000 0000 -> 0
0b1111 1111 & 0b0111 1111 = 0b0111 1111 -> 127

So the overall index is

size_t compute_index(string obj, size_t table_size){
    int hash = hashCode(obj);
    hash &= 0x7FFFFFFF;//this constant depend on character size
    hash %= table_size;
    return hash;
}

load factor α=\alpha =α= number of entries / total size of array larger load factor indicates, that we might need to increase size of the table array and rehash the entries.

Technieques For Handling Collisions​

Open addressing​

when we face a collision, during insertion lets say at index kkk, and if there's not a value there, we put the extra entry there. Now when finding a key, we have to compare the key to find and the key at the computed index and if they do not match, we look at indexes that come after that indexes in same as we did during the insertion. When the compared keys are identical, we return that entry. When deleting an entry which is not at the computed index and rather at a later index, we put a flag at that entry, so that when we search again, we indicate to keep looking further. Its importent to use the same formula to choose the next index in these operation. There are multiple techniques for choosing which indexes to look at.

  • linear probing hash_idx -> hash_idx + 1 -> hash_idx + 2
  • quadratic probling: hash_idx -> hash_idx 2 -> hash_idx 4
  • double hashing:
    • two different hash functions, second hash function must not return 0
    • in case of collision: hash_idx -> hash_idx + second_hash_val -> hash_idx + second_hash_val*2
  • in open addressing , we have to consider resizing the table array once load factor is about 0.6-0.7(resizing and rehashing is amortized O(1))

Chaining​

  • most used
  • chaining: array of linked lists, instead of simple array, for computed hash index hash_idx we store the entry in the linked list at index hash_idx
  • using chaining, we can have larger load factor. we can consider rehashing when the load factor is about one.
  • for rehashing, we'll have to create new array of linked list and then copy the entries while using the new table size for modulo operation during calculation of hash index
